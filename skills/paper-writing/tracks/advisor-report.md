# Track F: 科研汇报 (Advisor Report)

向导师汇报科研进展的内部文档。非正式发表，但质量决定导师对你的信任度和推进效率。

## 核心目标

科研汇报的目的不是"展示你做了什么"，而是：

1. **让导师快速判断方向是否正确**
2. **暴露当前的困难和决策点**
3. **争取导师的时间和指导投入到最需要的地方**

## 汇报模板

```markdown
# [项目名称] 科研进展汇报
日期: YYYY-MM-DD

## 一句话进展
[本周/本阶段最重要的一句话总结]

## 关键成果
- 成果1: [具体描述 + 数据/图表引用]
- 成果2: [具体描述]

## 遇到的问题
| 问题 | 尝试过的方案 | 结果 | 需要导师输入？ |
|------|-------------|------|---------------|
| ... | ... | ... | ✅/❌ |

## 下一步计划
| 任务 | 预计时间 | 依赖/风险 |
|------|----------|-----------|
| ... | ... | ... |

## 需要讨论的决策点
1. [决策1]: 方案A vs 方案B，我的倾向是A因为[理由]
2. [决策2]: ...

## 附件
- [实验结果图表]
- [代码链接]
```

## 导师原则适配

### 不适用的原则
- P1（引用分级）：内部汇报不需要严格引用
- P5-P8（热度/顶校/大厂）：不需要论证问题重要性
- P11（技术包装）：对导师要坦诚，不需要包装

### 适用的原则
- **P2（综合归纳）**：实验结果要归纳总结，不是堆数据
- **P4（理论问题）**：要指出理论上的困惑和空白
- **P9-P10（问题→方案对应）**：遇到的问题和解决方案要一一对应

## 写作要点

### 导师最看重的内容

1. **你的判断力**：不只是报告事实，还要给出你的分析和判断
2. **问题暴露**：及早暴露问题比独自挣扎一周后承认失败好得多
3. **资源请求的精准性**：不是"我需要帮助"，而是"关于X，我有两个方案，需要您帮我判断"

### 格式原则

- **先结论后过程**：一句话说最重要的结论，再展开
- **问题要具体**：不是"实验不太好"，而是"在dataset A上accuracy只有72%，分析发现原因可能是..."
- **图表说话**：一张好的实验结果图比三段文字更有说服力
- **标注需要导师输入的部分**：导师时间有限，帮他快速定位到需要他做决策的地方

### 常见错误

| ❌ 错误 | ✅ 正确 |
|---------|---------|
| 只报告做了什么，不分析结果 | 做了什么 + 结果意味着什么 + 下一步建议 |
| 隐藏负面结果 | 坦诚报告 + 分析原因 + 提出调整方案 |
| 列出 20 个实验的详细数据 | 归纳出 2-3 个关键发现，详细数据放附件 |
| "我需要帮助" | "关于X有两个方案，我倾向A，理由是..." |
| 模糊的下一步计划 | 有具体时间线和可衡量目标的计划 |

## 汇报频率建议

| 阶段 | 频率 | 重点 |
|------|------|------|
| 选题/调研 | 每周 | 文献发现 + 方向判断 |
| 方法设计 | 每周 | 设计决策 + 理论问题 |
| 实验阶段 | 每周或更频繁 | 实验结果 + 问题排查 |
| 论文撰写 | 每轮修改后 | 章节draft + 定位确认 |
| 投稿前 | 按需 | 最终检查 + venue 选择 |

## 质量检查

- [ ] 一句话进展清晰概括了核心状态
- [ ] 问题已暴露，不是藏着
- [ ] 需要导师决策的点明确标出
- [ ] 下一步计划有时间线
- [ ] 实验数据有归纳总结，不是数据堆砌
